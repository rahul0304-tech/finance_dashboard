Tech Stack Rationale & Architecture

This backend operates on a standard Model-View-Controller (MVC) logic utilizing modular frameworks specifically chosen to satisfy assignment rubrics flawlessly:

  • Django 6.0: Built-in AbstractUser modularity enabled clean, direct mapping of the required explicit string roles (VIEWER, ANALYST, ADMIN) securely at the database root without arbitrary one-to-many tables.
  • Django REST Framework (DRF): Severely reduces boilerplate codebase mappings when generating functional JSON serialized endpoints. Specifically, ViewSet permissions logic cleanly enforces granular conditional locks.
  • DRF-Spectacular: Actively replaces the deprecated drf-yasg module. Spectacular seamlessly maps OpenAPI Schema V3 structural generations, easily inferring JWT payload logic to parse dynamic endpoint categories correctly.
  • Aggregate Pipelines: Requesting aggregate endpoint lists avoids generating large dictionaries. Dashboard targets utilize native Django ORM aggregations (Sum(), Count()), maximizing query speeds reliably.

Core Requirements Addressed

  • Role Management: Developed a custom user model natively supporting strict ADMIN, ANALYST, and VIEWER roles.
  • Financial Records: Implemented comprehensive RESTful models to log monetary transactions efficiently.
  • Soft Deletion: Overriden the default Django DELETE framework methods to safely toggle an is_deleted boolean rather than permanently wiping critical database records.
  • Security: All endpoints are structurally protected by JSON Web Tokens (JWT) using HttpOnly cookies to prevent generic XSS payload vulnerability pipelines.

Role-Based Access Control Implementation

Role constraints are strictly enforced at the API ViewSet mapping levels using customized Django REST Framework permission classes protecting data routing parameters.

Defined Role System Capabilities
VIEWER Can only securely access aggregate statistics via /api/dashboard/. The API utilizes ORM queries to calculate totals without exposing raw transactional lists.
ANALYST Inherits all generic VIEWER properties identically, but additionally possesses dynamic GET access parsing full FinancialRecord querysets natively to examine line-item details. Structurally prevented from mutating schemas.
ADMIN Granted complete execution CRUD capabilities. Can manage user statuses conditionally (toggle role activation maps, overwrite passwords safely) and securely mutate explicit transaction records.

Database Schema & Object Modeling

Model: User (Subclassed AbstractUser)

username: CharField (Unique Identifier)
email: EmailField (Standard Payload)
role: CharField (Strict Scope Choices: 'ADMIN', 'ANALYST', 'VIEWER')
is_active: BooleanField
is_superuser: BooleanField

Model: FinancialRecord

amount: DecimalField
transaction_type: CharField (Options: 'INCOME', 'EXPENSE')
category: CharField
date: DateField
is_deleted: BooleanField (Default Boolean: False)

Note on Soft Delete Architecture: A customized generic Manager wrapper deployment automatically filters out is_deleted=True nodes explicitly from standard querysets (e.g. FinancialRecord.objects.all()), ensuring integrity mapping seamlessly without data destruction.

Input Validation & Error Handling

To ensure systemic payload integrity dynamically, generic constraints were abandoned in favor of unified strict native validation pipelines:

  • Input Validation: Dedicated validators explicitly inside DRF serializers confirm that categorical inputs are not left as whitespace, and that financial aggregate amounts are structurally validated strictly to remain positive values.
  • Unified Error Schemas: A global custom exception handler explicitly catches payload rejections globally, rewriting native Django tracebacks into a structured, standard JSON block identifying the exact boolean success state and uniform HTTP status_code.

Authentication System Logic

This backend incorporates advanced authorization infrastructures. SimpleJWT was actively leveraged mapping standard verification. To circumvent generic implicit frontend XSS vulnerabilities, the specific CookieTokenObtainPairView intercepts the internal layout, injects the sensitive refresh token directly into an HttpOnly scoped header, and returns simply the generic access token conditionally to the REST payload.
